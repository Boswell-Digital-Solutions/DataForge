## 🏗️ ARCHITECTURAL DECISIONS (CRITICAL)

### Database Architecture
**DECISION:** Shared PostgreSQL database (DataForge's database)
- All services write telemetry to same `events` table
- Single-user system (Charles Boswell only)
- Localhost-only, no authentication needed
- Focus on SQL injection prevention via parameterized queries

### Security Model
**DECISION:** Single-user, localhost-only
- No 2FA (unnecessary for single user)
- No separate database credentials (no security benefit)
- Command Central runs as Tauri app (zero network ports)
- SSH tunnel for remote access if needed

### Application Framework
**DECISION:** Tauri desktop application (not web app)
- Zero network exposure
- System tray integration
- Native notifications
- ~30MB RAM vs ~200MB for web
- Sub-100ms startup

### Telemetry Library
**DECISION:** Shared Python package `forge-telemetry`
- Used by DataForge, NeuroForge, and Rake
- Direct PostgreSQL connection
- Async-first API
- Every operation emits structured events

## 📊 TELEMETRY SCHEMA (CRITICAL)

### Events Table Structure
```sql
CREATE TABLE events (
  event_id UUID PRIMARY KEY,
  timestamp TIMESTAMPTZ NOT NULL,
  service VARCHAR(50) NOT NULL,              -- 'dataforge' | 'neuroforge' | 'rake'
  event_type VARCHAR(100) NOT NULL,          -- 'query', 'model_request', 'job_completed'
  severity VARCHAR(20) NOT NULL,             -- 'info' | 'warning' | 'error' | 'critical'
  tenant_id VARCHAR(100),                    -- For future multi-tenancy
  correlation_id UUID,                       -- For distributed tracing
  metadata JSONB,                            -- Service-specific context
  metrics JSONB,                             -- Performance metrics
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

### Required Event Types by Service

**DataForge Events:**
- `query` - Semantic search query completed
- `ingestion` - Documents added to collection
- `query_error` - Query failed

**NeuroForge Events:**
- `model_request` - LLM request completed
- `model_error` - LLM request failed

**Rake Events:**
- `job_started` - Ingestion job started
- `job_completed` - Job finished successfully
- `job_failed` - Job failed
- `phase_completed` - Individual stage completed (fetch, clean, chunk, embed, store)

### Git Commit Format
```

Types:
- feat: New feature
- fix: Bug fix

- style: Formatting, no code change
- refactor: Code refactoring
- test: Adding tests
- chore: Maintenance

Examples:
```

### Code Standards

**Python:**
- Type hints required (mypy strict mode)

- Clippy warnings as errors
- rustfmt for formatting
- tokio for async runtime
- sqlx for database access (compile-time query checking)

**TypeScript/Svelte:**
- Strict mode enabled
- ESLint + Prettier
- Vitest for testing
- TailwindCSS for styling

### Testing Requirements
- **Unit Tests:** 80%+ coverage for business logic
- **Integration Tests:** All API endpoints

- **E2E Tests:** Critical user flows (Playwright)
